### Create Test Harness

Create `tests/cucumber_tui.rs`:

```rust
use cucumber::{given, when, then, World};
use ratatui::backend::TestBackend;
use ratatui::Terminal;
use tark::tui_new::{TuiApp, AppState};

/// Test world that renders to a REAL TestBackend
#[derive(Debug, World)]
#[world(init = Self::new)]
pub struct TuiWorld {
    /// The actual TUI application
    app: TuiApp<TestBackend>,
    /// Terminal dimensions
    terminal_size: (u16, u16),
}

impl TuiWorld {
    fn new() -> Self {
        let backend = TestBackend::new(80, 24);
        let terminal = Terminal::new(backend).unwrap();
        Self {
            app: TuiApp::new(terminal),
            terminal_size: (80, 24),
        }
    }
    
    /// Render the app and return the buffer for verification
    fn render(&mut self) -> &ratatui::buffer::Buffer {
        self.app.render().unwrap();
        self.app.terminal().backend().buffer()
    }
    
    /// Check if a specific character exists at position
    fn char_at(&mut self, x: u16, y: u16) -> &str {
        let buffer = self.render();
        buffer.cell((x, y)).map(|c| c.symbol()).unwrap_or("")
    }
}

#[tokio::main]
async fn main() {
    TuiWorld::cucumber()
        .max_concurrent_scenarios(1)
        .run("tests/visual/tui/features/")
        .await;
}
```

### ‚ö†Ô∏è CRITICAL: Step Definitions Must Test REAL Rendering

**WRONG - Mock that always passes:**
```rust

    assert!(true);  // ‚ùå USELESS - always passes!
}
```

**RIGHT - Actually verify rendered output:**
```rust
#[then("I should see the terminal header at the top")]
async fn see_header(world: &mut TuiWorld) {
    // ‚úÖ Render and check actual buffer content
    let buffer = world.render();
    
    // Check for rounded corner at top-left
    assert_eq!(buffer.cell((0, 0)).unwrap().symbol(), "‚ï≠",

    // Check header text exists

        .map(|x| buffer.cell((x, 1)).unwrap().symbol())
        .collect();
    assert!(header_line.contains("Tark"), 
        "Header should contain agent name");
}
```

### Understanding Step Definitions

Step definitions link Gherkin steps to Rust code that tests REAL rendering:

```rust
// tests/visual/tui/features/step_definitions/common_steps.rs

#[given("the TUI application is running")]
async fn app_running(world: &mut TuiWorld) {
    // App is initialized in TuiWorld::new()
    // Verify it can render without errors
    world.render();
}

#[when(regex = r#"I type "(.+)""#)]
async fn type_text(world: &mut TuiWorld, text: String) {
    // Send actual key events to the app
    for c in text.chars() {
        world.app.handle_key(KeyCode::Char(c));
    }
}

#[then("I should see the terminal header at the top")]
async fn see_header(world: &mut TuiWorld) {
    let buffer = world.render();
    // Actually verify the header is rendered
    assert_eq!(buffer.cell((0, 0)).unwrap().symbol(), "‚ï≠");
}
```

### Adding New Step Definitions

When you encounter an unimplemented step:

1. Open `common_steps.rs`
2. Add the step with appropriate attribute (`#[given]`, `#[when]`, `#[then]`)
3. Implement the step logic **that tests REAL rendering**
4. Wire it to the `TuiWorld` state

Example:

```rust
#[then(regex = r"the status bar should show (.+)")]
async fn status_bar_shows(world: &mut TuiWorld, expected: String) {
    let buffer = world.render();
    let height = world.terminal_size.1;
    
    // Read the status bar line (last line)
    let status_line: String = (0..world.terminal_size.0)
        .map(|x| buffer.cell((x, height - 1)).unwrap().symbol())
        .collect();
    
    assert!(status_line.contains(&expected), 
        "Expected status bar to contain '{}', got '{}'", expected, status_line);
}
```

**Annotation Types:**

| Annotation | Purpose | Example |
|------------|---------|---------|

| `@ratatui-style:` | Styling (colors, modifiers) | `fg(Color::Rgb(...))`  |
| `@ratatui-state:` | Required state fields | `input: String` |
| `@ratatui-events:` | Event handlers needed | `KeyCode::Char ‚Üí handle_input()` |
| `@ratatui-behavior:` | How element should behave | Scrolling, toggling, etc. |
| `@ratatui-pattern:` | Code example reference | Section in RATATUI_MAPPING.md |

**How to use annotations:**

1. **Read the React component** (e.g., `Terminal.tsx`)
2. **Find @ratatui-* comments** near each UI element
3. **Use them as implementation hints** for your Rust code
4. **Cross-reference with RATATUI_MAPPING.md** for full examples
5. **Implement in Rust** following the guidance

**Example workflow:**

```typescript
// In Terminal.tsx (React):

  {/* @ratatui-widget: Paragraph with horizontal Layout */}
  {/* @ratatui-layout: Constraint::Length(2) */}

  <ModelDisplay model={currentModel} />

## ‚ö†Ô∏è Critical Guidelines (TDD Enhanced)

### DO: Test-First Development
- ‚úÖ **ALWAYS start with RED test** - Run cucumber test, verify it fails
- ‚úÖ **Write minimal code** - Just enough to make test pass (GREEN)
- ‚úÖ **Refactor with confidence** - Tests ensure nothing breaks
- ‚úÖ **Commit baselines** - E2E snapshots go into git for regression detection
- ‚úÖ **Run tests frequently** - After every small change
- ‚úÖ **Use feature files as spec** - They define ALL requirements (IMMUTABLE)
- ‚úÖ **Follow @ratatui-* annotations** - React files have implementation guides
- ‚úÖ **Compare snapshots** - Visual accuracy is mandatory
- ‚úÖ **Read React source** - Understand behavior from code + annotations
- ‚úÖ **Follow exact colors** - Use RGB values from theme.css and @ratatui-style
- ‚úÖ **Handle edge cases** - Empty lists, long text, narrow terminals

### DON'T: Anti-Patterns
- ‚ùå **Write code before test** - Violates TDD principle
- ‚ùå **Skip test verification** - "It looks right" is not enough
- ‚ùå **Ignore failing tests** - Fix immediately, don't accumulate debt
- ‚ùå **MODIFY FEATURE FILES** - They are IMMUTABLE acceptance criteria!
- ‚ùå **Ignore @ratatui-* annotations** - They tell you HOW to implement
- ‚ùå **Approximate visuals** - Snapshots must match reference exactly
- ‚ùå **Skip E2E tests** - Unit tests alone don't catch visual regressions
- ‚ùå **Commit without baselines** - Baselines are part of the code
- ‚ùå **Use wrong colors** - RGB values from @ratatui-style annotations are exact
- ‚ùå **Implement ahead** - Follow phase order, respect test gates
- ‚ùå **Skip React annotations** - Missing critical implementation details

### Test-Driven Mindset

```
‚ùå WRONG: "I'll write the feature, then add tests later"
‚úÖ RIGHT: "Let me read the feature file, run the test (RED), then implement"

‚úÖ RIGHT: "Tests pass AND snapshot matches reference - now ship it"

‚úÖ RIGHT: "Even small diffs indicate a problem - investigate and fix"

‚ùå WRONG: "I'll batch all features then test"
‚úÖ RIGHT: "One feature at a time, test-first, gate before proceeding"

‚ùå WRONG: "This feature file scenario is wrong, let me fix it"

## üÜò Troubleshooting (TDD Edition)

### "Cucumber test fails with 'step not found'"
‚Üí The step definition doesn't exist yet
‚Üí Open `tests/visual/tui/features/step_definitions/common_steps.rs`
‚Üí Add the step with `#[given]`, `#[when]`, or `#[then]` attribute
‚Üí Implement the step logic
‚Üí Wire it to `TuiWorld` state

### "Test passes but shouldn't (false positive)"
‚Üí The test is not asserting correctly
‚Üí Review the step definition implementation
‚Üí Add more specific assertions
‚Üí Check that you're testing actual app state, not just mocks

### "E2E snapshot doesn't match reference"
‚Üí **First:** Review the visual diff in `tests/visual/tui/diffs/`
‚Üí If diff is intentional (new feature): Update reference screenshot
‚Üí If diff is unintentional: Fix implementation to match reference
‚Üí Run `./tui_e2e_runner.sh --verify` after fix

### "asciinema/agg not found"
‚Üí Run `./tests/visual/tui_e2e_runner.sh --install-deps`
‚Üí Ensure `~/.cargo/bin` is in PATH for agg
‚Üí On Linux: Install font packages for proper rendering

### "Test hangs or times out"
‚Üí Check for missing `expect eof` in expect scripts
‚Üí Increase timeout in expect script: `set timeout 60`
‚Üí Verify TUI has an exit path (`/exit` command or Ctrl+C)

### "Colors don't match screenshot"
‚Üí Check RGB values in theme.css match exactly
‚Üí Verify terminal supports true color: `echo $COLORTERM`
‚Üí Set `COLORTERM=truecolor` if needed
‚Üí Compare hex codes: screenshot vs implementation

### "Feature file scenario is ambiguous"
‚Üí Read the scenario's `@tag` for context
‚Üí Check RATATUI_MAPPING.md for the component section
‚Üí Look at reference screenshot mentioned in feature file
‚Üí If still unclear: Note it, implement best interpretation, verify in review

### "I can't find a function in the Ratatui codebase"
‚Üí Use `grep -r "fn function_name"` or similar search
‚Üí Look for related functionality with different names
‚Üí Check if it's in a trait implementation
‚Üí Document as "NONE" and create it per RATATUI_MAPPING.md

### "The React code doesn't match the annotations"
‚Üí **Trust the React code**, not the annotations
‚Üí Feature files take precedence over both
‚Üí Document the discrepancy
‚Üí Implement based on: Feature file > React code > Annotations

### "The screenshot looks different from the code"
‚Üí **The screenshot is the source of truth**
‚Üí Check if it's a theme-specific appearance
‚Üí Look for dynamic styling based on state
‚Üí Implement what you see in the screenshot

### "All tests pass but visual doesn't match"
‚Üí This indicates incomplete test coverage
‚Üí Add more specific visual assertions to cucumber tests
‚Üí Ensure E2E snapshots are being compared correctly
‚Üí Review step definitions for correctness

### "I don't understand a Ratatui concept"
‚Üí Check RATATUI_MAPPING.md for examples
‚Üí Read official Ratatui documentation: https://ratatui.rs/
‚Üí Look at existing code in the codebase
‚Üí Feature files show the WHAT, RATATUI_MAPPING shows the HOW

## üí° TDD Philosophy

**Remember these principles:**

1. **Tests define requirements** - Feature files ARE the spec
2. **Red before green** - Always see the test fail first
3. **Small steps** - One scenario at a time
4. **Refactor fearlessly** - Tests protect you
5. **Snapshots verify visuals** - Not just "looks good to me"
6. **Commit baselines** - They're part of the codebase
7. **Respect gates** - Don't skip ahead

**The TDD cycle ensures:**
- ‚úÖ You build exactly what's needed (no more, no less)
- ‚úÖ You know when you're done (all tests green)
- ‚úÖ You catch regressions immediately (baselines)
- ‚úÖ You can refactor safely (tests as safety net)
